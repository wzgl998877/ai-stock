## 执行摘要

面向A股散户的缠论策略监控模块，自动识别自选股的缠论买卖点信号，在K线图和自选股列表上直观标注，让不懂缠论的普通用户也能"看到信号、自己再确认"。

**核心定位**：不是自动交易，不是替代用户判断，而是**减少用户手工盯盘的工作量**，让缠论信号自动呈现在已有的K线图上。

**两类用户**：
- **主用户**（不懂缠论）：看到信号提示 → 结合K线形态自己确认 → 决策
- **配置用户**（懂缠论的伙伴）：了解系统实现逻辑，验证信号准确性，未来可扩展配置

与通达信相比，本模块的差异化在于：**深度集成到个人知识库（模块一）和行情展示（模块二），信号 + 历史分析 + 当前行情一体化呈现**。

## 问题陈述

### 用户痛点
散户每天需要逐一打开自选股K线，用眼睛判断是否出现买卖信号，耗时且容易遗漏。缠论有明确的顶底分型规则，完全可以自动化。

### 现有工具的痛点
- 通达信有缠论插件，但与个人分析日志完全割裂
- 通达信的自定义公式有一定门槛，普通用户难以独立使用
- 没有与行业事件分析（模块一）的联动

- 当一笔向下结束（出现底部分型确认）→ 生成**买点信号**（绿色▲）

- [ ] 列表顶部可按信号状态筛选：全部 / 仅绿灯 / 仅红灯
- [ ] 每30分钟信号灯自动刷新（交易时段），页面无需手动刷新
- [ ] 非交易时段（收盘后/节假日）显示上一个交易日最终信号

## 功能需求

### 核心功能

**功能 1：K线合并算法**
- 依据缠论规则合并相邻K线（处理包含关系）：
  - 上升过程中，后一根K线被前一根包含 → 取两根K线的最高高点和最高低点
  - 下降过程中，后一根K线被前一根包含 → 取两根K线的最低高点和最低低点
- 合并结果存入中间表，作为后续顶底分型识别的输入
- 合并后的K线序列需保留原始交易日期映射关系

**功能 2：顶底分型识别**
- **顶部分型**：连续3根合并K线，中间一根的最高价高于左右两根 → 标记为顶部分型
- **底部分型**：连续3根合并K线，中间一根的最低价低于左右两根 → 标记为底部分型
- 相邻同向分型之间必须有至少1根独立K线（非包含关系），否则取极值更高/低的那个

**功能 3：笔的识别**
- 顶底分型交替排列（顶-底-顶-底...）形成笔
- 一笔的成立条件：顶部分型到底部分型之间（或反向）至少有5根合并K线（含端点）
- 笔的方向：顶部分型 → 底部分型 = 向下的笔；底部分型 → 顶部分型 = 向上的笔

**功能 4：信号生成**
- 向下的笔结束（出现底部分型确认）→ 生成**买点信号**
- 向上的笔结束（出现顶部分型确认）→ 生成**卖点信号**
- 信号确认时机：底部分型的第三根K线收盘后确认（避免未收盘K线的噪音信号）
- 信号写入 `strategy_signals` 表，包含：股票代码、信号类型、触发日期、触发收盘价

**功能 5：定时计算任务**
- 使用 **APScheduler** 实现定时任务
- 计划任务：交易时段内每30分钟触发一次（9:30 / 10:00 / 10:30 ... / 14:30 / 15:00）
- 节假日和周末不触发（通过AKShare交易日历判断）
- 每次任务计算当前用户所有自选股的最新信号
- 计算结果写入Redis缓存（Key: `signal:{stock_code}`，TTL：1小时）
- 同步写入PostgreSQL持久化（历史信号记录）

**功能 6：信号状态API**
- `GET /api/strategy/{code}/signal`：返回指定股票当前信号状态和最近信号时间
- `GET /api/strategy/{code}/history`：返回指定股票历史信号列表
- `POST /api/strategy/recalculate`：手动触发全量重算（配置用户使用）
- 模块二 K线页直接调用上述API获取标注数据

## 技术约束

### 性能
- 单只股票信号计算时间：≤1秒（3年日线数据约750根K线）
- 100只自选股全量计算：≤30秒（可并行计算）
- Redis缓存命中率：信号状态接口 >95%（缓存TTL 1小时）

### 数据
- 信号计算依赖日线历史数据，从模块二的 `stock_kline_daily` 表读取，不重复拉取
- 最少需要 **30个交易日** 的数据才能开始计算（不足则标记灰色）
- 信号历史表 `strategy_signals` 只存已确认信号（第三根K线已收盘），不存未确认中间状态

### 集成
- **模块二集成**：K线图箭头标注通过前端调用 `/api/strategy/{code}/history` 实现，ECharts markPoint组件渲染
- **模块二自选股**：信号灯数据通过 `/api/strategy/{code}/signal` 接口获取，随自选股列表一并加载
- **数据复用**：共用模块二的K线历史数据，不重复存储

### 技术栈
- 计算引擎：Python（纯逻辑，不依赖外部库，便于单元测试）
- 定时任务：**APScheduler**（集成在FastAPI进程内）
- 缓存：Redis（信号状态）
- 持久化：PostgreSQL（历史信号记录）
- K线标注渲染：ECharts 5 `markPoint` 组件

## 风险评估

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 缠论算法实现有偏差（与标准理论不一致） | 高 | 高 | 开发完成后，由懂缠论的伙伴用历史K线逐一对照通达信信号验证；建立单元测试用例 |
| AKShare数据缺失导致计算异常 | 中 | 中 | 计算前检查数据完整性，数据不足的股票标记灰色而非报错 |
| 未收盘K线产生噪音信号 | 中 | 中 | 严格只在K线确认收盘后生成信号，交易时段内的实时K线不纳入计算 |
| 并发计算100只股票时服务器资源不足 | 低 | 中 | 使用异步并发（asyncio），控制并发数 ≤10，避免同时拉取过多数据 |
| 自选股数量过多（>200只）时计算超时 | 低 | 低 | 设置单次任务最大计算200只，超出的排队到下一个时间片 |

### 数据库表结构

```sql
-- 缠论信号历史表
CREATE TABLE strategy_signals (
    id SERIAL PRIMARY KEY,
    stock_code VARCHAR(10) NOT NULL,
    signal_type VARCHAR(10) NOT NULL,  -- 'buy' | 'sell'
    signal_date DATE NOT NULL,
    trigger_price DECIMAL(10,2),
    period VARCHAR(20) DEFAULT 'daily',
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_signals_code_date ON strategy_signals(stock_code, signal_date DESC);
```
